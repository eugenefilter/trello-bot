# Как работает Trello Bot

## Общая схема

Telegram-сообщение → Webhook → Job → Правило маршрутизации → Карточка в Trello

---

## Настройка (один раз)

### 1. Подключить Trello-доску

В `Трело - подключение (/admin/trello-connections)` создать подключение — указать название, ID доски, API Key и Token.
После создания нажать **Синхронизировать** — это загрузит в базу все списки, метки и участников доски.

### 2. Настроить правила маршрутизации

В `Бот - Правила маршрутизации (/admin/routing-rules)` создать правило. Правило отвечает на вопрос: «какое сообщение → в какой список попадает».

---

## Правила маршрутизации — подробно

### Как работает выбор правила

При каждом входящем сообщении система перебирает все активные правила **от большего приоритета к меньшему** и берёт первое подходящее. Если ни одно не подошло — сообщение игнорируется.

### Условия

Все условия опциональны. Незаполненное условие означает «подходит любое значение». Если заполнены несколько условий — все они должны совпасть одновременно.

| Поле | Что проверяет |
|---|---|
| **Chat ID** | Только сообщения из конкретного чата |
| **Тип чата** | `private` — личка, `group` — группа, `supergroup` — супергруппа |
| **Команда** | Текст начинается с `/task`, `/bug` и т.д. |
| **Только с фото** | Включено — только сообщения с фото, выключено — не проверяется |

Если **все условия пустые** — правило срабатывает на любое сообщение (catch-all).

### Приоритет

Число. Чем больше — тем раньше проверяется. Рекомендуемая шкала:

- `100` — самые специфичные правила (конкретный чат + команда)
- `50` — правила по команде или типу чата
- `10` — правила по типу контента (фото)
- `0` — catch-all (срабатывает если ничего другого не подошло)

**Важно:** catch-all с приоритетом `0` нужно создавать последним — иначе он перехватит все сообщения раньше специфичных правил.

### Действие при срабатывании

- **Целевой список** — в какую колонку Trello создать карточку
- **Метки** — какие цветные метки назначить
- **Участники** — кого назначить ответственным
- **Шаблон заголовка** и **описания** — текст карточки с подстановкой переменных

## Примеры правил

### Пример 1 — Catch-all (любое сообщение в «Входящие»)

Самое простое правило. Все сообщения из любого источника попадают в один список.

| Поле | Значение |
|---|---|
| Название | Все входящие |
| Chat ID | — |
| Тип чата | — |
| Команда | — |
| Только с фото | — |
| Приоритет | `0` |
| Целевой список | Входящие |
| Заголовок | `{{first_name}}: {{text_preview}}` |
| Описание | `От: {{first_name}} (@{{username}})`<br>`Дата: {{date}} {{time}}`<br><br>`{{text}}` |

---

### Пример 2 — Команда `/bug` создаёт карточку в «Баги»

Пользователь пишет `/bug Кнопка не работает` — карточка уходит в список «Баги» с меткой «Ошибка» и назначается на тимлида.

| Поле | Значение |
|---|---|
| Название | Баг-репорт |
| Chat ID | — |
| Тип чата | — |
| Команда | `/bug` |
| Только с фото | — |
| Приоритет | `50` |
| Целевой список | Баги |
| Метки | Ошибка (red) |
| Участники | @teamlead |
| Заголовок | `🐛 {{text_preview}}` |
| Описание | `Автор: {{first_name}} (@{{username}})`<br>`Дата: {{date}} {{time}}`<br><br>`{{text}}`<br><br>`Цитируемое сообщение:`<br>`{{reply_text}}` |

---

### Пример 3 — Фото из группы идёт в «Медиа-архив»

Любое сообщение с фото из группового чата создаёт карточку в отдельный список.

| Поле | Значение |
|---|---|
| Название | Фото из группы |
| Chat ID | — |
| Тип чата | `group` |
| Команда | — |
| Только с фото | ✅ Да |
| Приоритет | `10` |
| Целевой список | Медиа-архив |
| Заголовок | `📷 {{first_name}} — {{date}}` |
| Описание | `Автор: {{first_name}} (@{{username}})`<br>`{{text}}` |

---

### Пример 4 — Конкретный чат с конкретной командой

Только из рабочей группы (Chat ID `-1001234567890`) команда `/task` создаёт задачу и назначает двух исполнителей.

| Поле | Значение |
|---|---|
| Название | Задача из рабочей группы |
| Chat ID | `-1001234567890` |
| Тип чата | — |
| Команда | `/task` |
| Только с фото | — |
| Приоритет | `100` |
| Целевой список | To Do |
| Метки | Задача (blue) |
| Участники | @alice, @bob |
| Заголовок | `✅ {{text_preview}}` |
| Описание | `Поставил: {{first_name}}`<br>`Дата: {{date}} {{time}}`<br><br>`{{text}}` |

### Как расставить приоритеты при комбинации правил

Пример: есть правило «Баг по команде» (приоритет `50`) и catch-all (приоритет `0`).

Пользователь пишет `/bug Не работает вход` → система проверяет правила от большего приоритета к меньшему → находит «Баг по команде» (50) → создаёт карточку в «Баги» → до catch-all не доходит.

Пользователь пишет `Просто текст` → правило «Баг» не подходит (нет `/bug`) → catch-all (0) срабатывает → карточка идёт в «Входящие».

## Шаблоны карточки — переменные

| Переменная | Что подставляется |
|---|---|
| `{{first_name}}` | Имя отправителя |
| `{{username}}` | @username (без @) |
| `{{user_id}}` | Telegram ID пользователя |
| `{{text}}` | Полный текст сообщения (команда `/bug` обрезается) |
| `{{text_preview}}` | Первые 80 символов |
| `{{date}}` | Дата в формате дд.мм.гггг |
| `{{time}}` | Время в формате чч:мм |
| `{{chat_type}}` | Тип чата (private / group / supergroup) |
| `{{reply_text}}` | Текст цитируемого сообщения (если есть reply) |

Неизвестные переменные остаются в тексте как есть — `{{unknown}}` не вызывает ошибку.

### Переменная `{{reply_text}}` — ответ на чужое сообщение

Когда пользователь вызывает команду `/bug` (или другую), отвечая на чужое сообщение, переменная `{{reply_text}}` содержит текст или подпись этого цитируемого сообщения. Фотографии из цитируемого сообщения также автоматически прикрепляются к карточке.

Если ответа нет — `{{reply_text}}` подставляется как пустая строка.

**Важно:** если ты используешь кастомный шаблон описания (не дефолтный), нужно явно добавить `{{reply_text}}` в шаблон — иначе текст цитаты не попадёт в карточку.

Рекомендуемый шаблон описания для правил с командой `/bug`:

```
Автор: {{first_name}} (@{{username}})
Дата: {{date}} {{time}}

Текст:
{{text}}

Цитируемое сообщение:
{{reply_text}}
```

## Как работает входящее сообщение

1. Telegram отправляет сообщение на webhook `/webhooks/telegram`
2. Сообщение сохраняется в БД и ставится в очередь
3. Job достаёт сообщение, ищет подходящее правило (от большего приоритета к меньшему)
4. Если правило найдено — рендерит шаблоны, создаёт карточку в Trello, при наличии фото — прикрепляет его к карточке
5. Пользователь получает ответ в Telegram со ссылкой на созданную карточку
6. Если правило не найдено — сообщение молча помечается как обработанное

---

## Мониторинг

- **Логи → Карточки Trello** — каждая попытка создать карточку (успех или ошибка с деталями)
- **Логи → Telegram сообщения** — все входящие сообщения, статус обработки

---

## Повторная синхронизация справочников

Если на доске добавились новые списки, метки или участники — нажать **Синхронизировать** в подключении. Удалённые из Trello элементы помечаются неактивными и не мешают существующим правилам.
